# コードレビューの受け方・書き方完全ガイド2026

コードレビューは、エンジニアの日常業務の中で最も重要な活動の一つです。レビューを通じてコード品質が向上するだけでなく、チーム全体の技術力が底上げされ、属人化も防げます。

しかし、多くのエンジニアが「レビューの仕方がわからない」「指摘をどう書けばいいかわからない」「指摘を受けると落ち込む」と感じています。本記事では、レビューの受け手と書き手、両方の視点から実践的なノウハウを解説します。

## コードレビューの目的を正しく理解する

### レビューの本質的な目的

```
コードレビューの目的:
├── ✅ バグの早期発見（本番障害を防ぐ）
├── ✅ コード品質の維持（読みやすさ・保守性）
├── ✅ 知識の共有（属人化の防止）
├── ✅ チームの技術水準の統一
├── ✅ ジュニアエンジニアの教育
└── ✅ 設計の妥当性確認

コードレビューの目的ではないもの:
├── ❌ 個人の能力を評価すること
├── ❌ マウントを取ること
├── ❌ 完璧なコードを求めること
└── ❌ コーディングスタイルの好みを押し付けること
```

### レビューの投資対効果

```
コードレビューのROI:
├── バグ修正コスト:
│   ├── 開発中に発見: 1x
│   ├── テスト中に発見: 5x
│   ├── 本番で発見: 30x
│   └── レビューで発見: 1.5x ← 最もコスパが良い
│
├── 知識共有の効果:
│   ├── レビューなし: 1人だけがコードを理解
│   └── レビューあり: 最低2人がコードを理解（バス係数の向上）
│
└── 教育効果:
    ├── レビューを受ける側: 実際のコードで学べる（座学より効果的）
    └── レビューする側: 説明する過程で自分の理解が深まる
```

## PRの出し方（レビューを受ける側）

### 良いPRの条件

```
理想的なPR:
├── サイズ: 200行以内（最大でも400行）
├── スコープ: 1つの機能 or 1つのバグ修正
├── テスト: 変更に対応するテストが含まれている
├── 説明: 何をなぜ変更したかが明記されている
├── スクリーンショット: UIの変更があれば添付
└── セルフレビュー: 自分でdiffを確認済み

避けるべきPR:
├── ❌ 1000行超の巨大PR
├── ❌ 複数の機能変更が混在
├── ❌ テストなし
├── ❌ 説明なし（タイトルのみ）
├── ❌ WIP状態での正式レビュー依頼
└── ❌ フォーマット変更と機能変更の混在
```

### PRテンプレート

```markdown
## 概要
<!-- この PR で何を変更するか、1-2行で説明 -->

## 変更理由
<!-- なぜこの変更が必要か -->
- 関連Issue: #XXX

## 変更内容
<!-- 技術的な変更の詳細 -->
-

## テスト
<!-- どのようにテストしたか -->
- [ ] ユニットテスト追加
- [ ] 手動テスト実施
- [ ] エッジケースの確認

## スクリーンショット
<!-- UIの変更がある場合 -->
| Before | After |
|--------|-------|
|        |       |

## レビューポイント
<!-- 特に見てほしい箇所 -->
-

## チェックリスト
- [ ] セルフレビュー完了
- [ ] テストが全てパス
- [ ] ドキュメント更新（必要な場合）
- [ ] マイグレーション不要 / 実行済み
```

### セルフレビューのチェックリスト

PRを出す前に、自分でこのチェックリストを確認しましょう。

```
セルフレビュー項目:
├── 基本チェック
│   ├── □ 不要なconsole.log / print文が残っていないか
│   ├── □ コメントアウトされたコードが残っていないか
│   ├── □ TODO/FIXMEコメントは意図的か
│   ├── □ 変数名・関数名は適切か
│   └── □ フォーマッターを通したか
│
├── ロジックチェック
│   ├── □ エッジケースを考慮しているか
│   ├── □ エラーハンドリングは適切か
│   ├── □ nullチェックは必要十分か
│   ├── □ 並行処理の問題はないか
│   └── □ パフォーマンスの問題はないか
│
├── セキュリティチェック
│   ├── □ SQLインジェクション対策
│   ├── □ XSS対策
│   ├── □ 認証・認可の確認
│   ├── □ 機密情報のハードコードなし
│   └── □ 入力バリデーション
│
└── テストチェック
    ├── □ 正常系テストがある
    ├── □ 異常系テストがある
    ├── □ 境界値テストがある
    └── □ テストが全てパスする
```

## レビューコメントの書き方（レビューする側）

### コメントの分類プレフィックス

レビューコメントには、重要度を示すプレフィックスを付けましょう。これにより、レビューイーが優先順位を判断しやすくなります。

```
プレフィックス一覧:
├── [MUST] → 修正必須。マージブロッカー
│   例: [MUST] SQLインジェクションの脆弱性があります
│
├── [SHOULD] → 強く推奨。修正しない場合は理由を説明
│   例: [SHOULD] この関数は責務が多すぎるので分割を推奨します
│
├── [IMO] → 個人的な意見。採用するかは任意
│   例: [IMO] この変数名は`userList`より`activeUsers`の方が意図が明確かも
│
├── [nit] → 些細な指摘。対応は任意
│   例: [nit] ここの空行は不要かもしれません
│
├── [Q] → 質問。コードの意図の確認
│   例: [Q] この条件分岐はどういうケースを想定していますか？
│
└── [PRAISE] → 良い点を褒める
    例: [PRAISE] このエラーハンドリングのアプローチ、すごく良いですね！
```

### 良いレビューコメントの書き方

```
❌ 悪いコメント:
「これダメ」
「ここ直して」
「なんでこう書いたの？」

→ 何がダメなのか、どう直すべきかが不明
→ 攻撃的に読めてしまう

✅ 良いコメント:
「[SHOULD] この箇所、N+1クエリになっています。
 `includes(:user)` を使うと1クエリで取得できます。

 ```ruby
 # Before
 posts.each { |p| p.user.name }

 # After
 posts.includes(:user).each { |p| p.user.name }
 ```

 参考: https://guides.rubyonrails.org/...」

→ 問題点 + 理由 + 改善案 + 参考リンク
```

### レビューコメントテンプレート

```markdown
## バグ指摘テンプレート
[MUST] ここに{{バグの内容}}があります。

**問題**: {{何が起きるか}}
**原因**: {{なぜ起きるか}}
**修正案**:
{{コード例}}

## 設計改善テンプレート
[SHOULD] {{現在の実装の問題点}}。

{{改善案の説明}}

```
// Before
{{現在のコード}}

// After（提案）
{{改善後のコード}}
```

**理由**: {{なぜこの改善が必要か}}

## 質問テンプレート
[Q] {{質問内容}}

{{背景・文脈の説明}}
この部分の意図を教えてもらえますか？
```

## レビュー指摘の受け方

### 心構え

```
レビュー指摘を受けるときの心構え:
├── 指摘はコードに対するもの、自分の人格に対するものではない
├── レビュアーは敵ではなく味方（コード品質向上という共通目標）
├── 「完璧なコードを1回で書ける人はいない」が大前提
├── 指摘が多い = 学習機会が多い（ポジティブに捉える）
└── わからない指摘は遠慮なく質問する
```

### 指摘への対応フロー

```
指摘を受けたら:
├── Step 1: 感情的にならず、まず内容を理解する
│   └── すぐに返信しない。5分置いてから読み直す
│
├── Step 2: 指摘の種類を判断する
│   ├── MUST → 即対応
│   ├── SHOULD → 理解した上で対応方針を決める
│   ├── nit/IMO → 可能なら対応、難しければ理由を説明
│   └── Q → 質問に回答する
│
├── Step 3: 対応する
│   ├── 修正する場合 → コードを修正してコメントで報告
│   ├── 同意しない場合 → 理由を丁寧に説明する
│   └── わからない場合 → 「もう少し詳しく教えていただけますか？」
│
└── Step 4: 学びを記録する
    └── レビュー指摘ログに追加（同じ指摘を受けないために）
```

### 返信のテンプレート

```
修正した場合:
「ご指摘ありがとうございます。修正しました。
 {{修正内容の簡単な説明}}」

同意しない場合:
「ご指摘ありがとうございます。
 この箇所は{{理由}}のため、現在の実装にしています。
 {{代替案があれば提示}}
 いかがでしょうか？」

わからない場合:
「ご指摘ありがとうございます。
 {{自分の理解}}と認識していますが、
 {{わからない部分}}について、もう少し詳しく教えていただけますか？」
```

## レビューの効率化テクニック

### レビュアーとして効率的にレビューする方法

```
効率的なレビュー手順:
├── Step 1: PRの説明を読む（2分）
│   └── 変更の目的・スコープを理解
│
├── Step 2: 全体のdiffをざっと見る（3分）
│   └── 変更の規模感・構造を把握
│
├── Step 3: テストコードから読む（5分）
│   └── 期待される振る舞いを理解
│
├── Step 4: メインのロジックを読む（10〜15分）
│   └── テストで理解した振る舞いと照合
│
├── Step 5: エッジケース・エラー処理を確認（5分）
│   └── テストでカバーされていない部分を指摘
│
└── Step 6: コメントをまとめて送信（3分）
    └── 一括送信で通知を減らす

所要時間の目安:
├── 〜100行: 15分
├── 100〜300行: 30分
├── 300〜500行: 45分
└── 500行超: 分割を依頼
```

### 自動化でレビュー負荷を減らす

```
自動化すべきレビュー項目:
├── コーディングスタイル → ESLint / Prettier / Rubocop
├── 型チェック → TypeScript / mypy
├── テストカバレッジ → Jest / pytest + CI
├── セキュリティ → Snyk / Dependabot
├── コード品質 → SonarQube / CodeClimate
├── パフォーマンス → Lighthouse CI
└── コミットメッセージ → commitlint

Linterで自動チェックすべき項目を人間がレビューするのは時間の無駄
→ CI/CDで自動化し、人間は設計・ロジック・ビジネス要件に集中
```

## チームのレビュー文化を改善する

### レビュー文化のアンチパターン

| アンチパターン | 症状 | 改善策 |
|-------------|------|-------|
| ゲートキーパー型 | 特定の人しかレビューしない | ローテーション制にする |
| 放置型 | PRが数日放置される | レビューSLA（24時間以内）を設定 |
| マウント型 | 高圧的な指摘が多い | コメントプレフィックスを導入 |
| 承認ゴム印型 | 内容を見ずにApprove | レビューチェックリストを導入 |
| 完璧主義型 | 些細な指摘で何往復もする | MUST/SHOULD/nitの区分を徹底 |

### レビュー文化のベストプラクティス

```
チームで合意すべきレビュールール:
├── レビューSLA: 依頼から24時間以内に初回レビュー
├── PRサイズ上限: 400行以内（超える場合は分割を依頼）
├── Approve条件: MUSTコメントが全て解消済み
├── レビュアー数: 最低1人（重要な変更は2人）
├── セルフレビュー: PR作成者が最初にセルフレビューすること
├── 褒める文化: 良いコードには [PRAISE] コメントを残す
└── ペアレビュー: 複雑な変更は画面共有でレビュー
```

## コードレビューでよくある議論と判断基準

### 命名に関する議論

```javascript
// 議論: リストの変数名
const data = fetchUsers();        // ❌ 抽象的すぎる
const list = fetchUsers();        // ❌ 型名が変数名
const userList = fetchUsers();    // △ 冗長（List不要）
const users = fetchUsers();       // ✅ シンプルで明確

// 議論: boolean の命名
const flag = true;                // ❌ 何のフラグかわからない
const isActive = true;            // ✅ is/has/can/should プレフィックス
const hasPermission = true;       // ✅
const canEdit = true;             // ✅
```

### エラーハンドリングに関する議論

```typescript
// 議論: try-catch の粒度

// ❌ 全体を try-catch で囲む（どこでエラーが起きたかわからない）
try {
  const user = await fetchUser(id);
  const orders = await fetchOrders(user.id);
  const report = generateReport(orders);
  await sendEmail(user.email, report);
} catch (error) {
  console.error(error);
}

// ✅ 適切な粒度で try-catch（エラーの種類ごとに対応）
const user = await fetchUser(id);
if (!user) {
  throw new NotFoundError(`User ${id} not found`);
}

let orders;
try {
  orders = await fetchOrders(user.id);
} catch (error) {
  throw new ServiceError('Failed to fetch orders', { cause: error });
}

const report = generateReport(orders);

try {
  await sendEmail(user.email, report);
} catch (error) {
  logger.warn('Email delivery failed', { userId: user.id, error });
  // メール送信失敗は致命的ではないのでログだけ残して続行
}
```

### テストに関する議論

```typescript
// 議論: モックの粒度

// ❌ 全てをモック（テストの意味が薄い）
jest.mock('./database');
jest.mock('./cache');
jest.mock('./logger');
jest.mock('./validator');

// ✅ 外部依存のみモック（ビジネスロジックは実際に動かす）
jest.mock('./database');  // DBアクセスのみモック
// validator, logger は実際のものを使う
```

## レビュースキルの成長ロードマップ

```
Level 1（レビューを受ける側）:
├── セルフレビューの習慣がある
├── 指摘を素直に受け入れて修正できる
├── 質問テンプレートを使って的確に質問できる
└── レビュー指摘をログに残して学習している

Level 2（基本的なレビューができる）:
├── コーディング規約違反を指摘できる
├── 基本的なバグを発見できる
├── テスト不足を指摘できる
├── プレフィックス（MUST/SHOULD/nit）を使い分けている
└── 褒めるコメントも書いている

Level 3（設計レベルのレビューができる）:
├── 設計の改善提案ができる
├── パフォーマンス問題を指摘できる
├── セキュリティリスクを発見できる
├── 代替案をコード付きで提示できる
└── レビューを通じてメンバーを育成している

Level 4（レビュー文化をリードできる）:
├── チームのレビュールールを策定・改善している
├── レビュー効率化の仕組みを導入している
├── 難しい設計判断についてチームで議論をリードしている
├── 新メンバーにレビューの仕方を教えている
└── レビュー品質の振り返りを定期的に行っている
```

## まとめ

コードレビューは、エンジニアチームの生産性と品質を大きく左右する活動です。

1. **PRは小さく・明確に**: 200行以内、1つの目的に絞る
2. **セルフレビューを徹底**: 自分でdiffを確認してからレビュー依頼
3. **コメントにはプレフィックスを**: MUST/SHOULD/nit/Q/PRAISEで優先度を明示
4. **指摘はコードに対するもの**: 人格攻撃ではない。学習機会として活用
5. **自動化できるものは自動化**: Linter/CI/CDに任せ、人間は設計に集中

レビュースキルは、経験を積むことで確実に向上します。最初は受ける側として学び、やがてレビュアーとしてチームに貢献していきましょう。
